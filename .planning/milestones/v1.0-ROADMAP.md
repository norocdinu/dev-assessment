## Core Value Delivered

A candidate receives a link and gets a fair, repeatable, automatically-graded test — every time, for any technology, at any seniority level.

---

## Key Accomplishments

1. **Admin auth + RBAC** — JWT httpOnly cookie sessions; Owner (full access) and Reviewer (read-only) roles enforced across API and UI
2. **Question bank CMS** — Full CRUD with immutable versioning (family_id + version + is_latest), soft-delete, audit log; bulk import/export CSV; bulk archive and hard-delete with reference protection
3. **Candidate test portal** — Zero-friction link entry (no login); seeded RNG for deterministic question sets; 30-min hybrid timer (client UI + server hard cutoff); page-refresh recovery via localStorage; auto-submit on expiry
4. **Auto-grading engine** — MCQ exact-match grading synchronous on submission; skill-area breakdown; full candidate answer sheet with correct/incorrect per question
5. **Admin dashboard** — Paginated submissions list with TanStack sort/filter; side-by-side candidate comparison; aggregate stats (avg score, pass rate, score distribution chart); CSV export
6. **Phase 5 improvements** — Server-side pagination across question bank and submissions; sonner toast feedback on all write ops; sidebar active-nav highlighting; DataTable pagination component; floating bulk action bar

### Phase 2 — Test Experience ✓ (2026-04-27)

**Goal**: A candidate receives a link, takes a 30-minute timed MCQ test, and submits. Nothing breaks.

**Requirements**: ASSESS-01, ASSESS-02, ASSESS-03, ASSESS-04, ASSESS-05, ASSESS-06, TESTS-02, TESTS-03, TESTS-04, TESTS-05

**Plans**: 5 (02-01 through 02-05)

**Deliverables**:

- Candidate portal: zero-friction entry, question display, MCQ answer input
- 30-min countdown timer: client-side UI (green/yellow/red stages) + server-side hard cutoff
- Auto-submit on timer expiry; server rejects late answers
- Page-refresh recovery: localStorage stores started_at + answers; resumes with correct remaining time
- Seeded question selection: same link → same questions always; different seeds → different sets

**Success Criteria — All Passed:**
1. Candidate opens link, sees first question within 3 seconds (no login prompt) ✓
2. Timer counts down live; turns red at 60 seconds ✓
3. Candidate refreshes mid-test and resumes with correct remaining time ✓
4. Timer reaches 0:00 → auto-submit → cannot answer further ✓
5. Two different links for same test config show different question sets ✓
6. Same link opened twice shows identical question set ✓

## Architecture at Ship

**Stack**: PostgreSQL + Redis | React/Next.js 14 | Fastify | TypeScript monorepo (npm workspaces)

**Key Decisions:**
| Decision | Rationale |
|----------|-----------|
| MCQ-only for v1 | Fast to build, easy to auto-grade, sufficient for screening |
| No candidate auth — link-based | Zero friction; interviewer controls context |
| Seeded RNG | Same link = same questions always; different candidates get equivalent fairness |
| Hybrid timer | Client UI for experience; server hard cutoff for integrity |
| Immutable question versioning | Past submissions forever linked to question version used |
| Owner + Reviewer RBAC | Team hiring: interviewers can view, only question owners can edit |

**DB Schema**: admin_users · technologies · questions (family_id + version + is_latest) · test_configs · candidate_links · submissions · candidate_answers · audit_log
